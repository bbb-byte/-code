# 工程质量与复现性答辩说明

## 1. 问题背景

本项目在早期实现公网任务时，存在典型的本机环境耦合问题，主要表现为：

- Python 解释器路径依赖作者本机安装位置
- Chrome / Chromium 浏览器路径依赖作者本机安装位置
- 浏览器 Profile 路径依赖 Windows 本地用户目录
- 后端在执行公网任务时默认假设脚本与运行依赖都在同一台机器、同一个环境中可直接访问

这类问题会直接影响以下几项工程指标：

- 跨机器运行能力
- Docker 复现能力
- 团队协作可移植性
- 后期部署与维护成本

## 2. 已识别的工程风险

如果保留硬编码实现，会出现以下风险：

1. 在 Linux / Docker 环境中，由于本机路径不存在，公网任务无法直接启动。
2. 在其他同学或评审老师电脑上，由于 Python、Chrome 安装路径不同，项目无法开箱即用。
3. 当后端容器未内置 Python、Node、Chromium 时，Docker 只能启动 Web 服务，无法真正执行公网采集任务。
4. 出现异常时，如果只返回底层报错，排障成本高，不利于工程维护。

## 3. 本次改进思路

本次改造遵循“先解耦配置，再拆分运行时”的原则，分两层处理。

### 3.1 可配置化

将原先硬编码的本机路径改为统一环境变量配置：

- `PUBLIC_TASK_PYTHON`
- `PUBLIC_TASK_BROWSER_PATH`
- `PUBLIC_TASK_BROWSER_CHANNEL`
- `PUBLIC_TASK_BROWSER_PROFILE_DIR`
- `PUBLIC_TASK_WORKER_URL`
- `PUBLIC_TASK_WORKSPACE_ROOT`

同时增加自动发现与兜底逻辑：

- Python 未显式配置时，自动尝试 `python3`、`python`、`python.exe`
- 浏览器路径未显式配置时，允许走 Playwright / 运行环境默认发现逻辑
- 浏览器 Profile 目录统一收敛到项目运行目录或容器挂载目录

这样做的意义是：

- 代码不再绑定作者个人电脑
- 配置与代码分离
- 同一套代码可在本机、他人电脑、Docker 中复用

### 3.2 容器职责拆分

为解决“后端镜像无法直接执行公网任务”的问题，引入独立 `public-task-worker` 服务：

- `backend`：负责接口、任务编排、结果入库
- `public-task-worker`：负责执行 Python / Node / Chromium 公网任务脚本
- `mysql` / `redis`：负责数据库与缓存

拆分后的好处：

- 后端服务与采集执行环境解耦
- Python、Node、Chromium 等重依赖集中在 `worker` 中维护
- Docker 中的公网任务具备独立运行能力
- 后续若需要扩展为多 worker 或远程执行，架构更容易演进

## 4. 当前已完成的改造点

### 4.1 路径硬编码清理

已将以下问题从主链路中移除：

- 硬编码 Python 解释器路径
- 硬编码 Chrome 可执行文件路径
- 硬编码 Windows 用户目录下的浏览器 Profile

### 4.2 环境变量统一入口

后端、Python 脚本、Node 脚本现已通过统一环境变量读取运行配置，避免各模块各自定义一套隐式规则。

### 4.3 Docker 化执行链打通

后端可通过 `PUBLIC_TASK_WORKER_URL` 将公网任务转发给 `worker` 执行，`worker` 镜像内提供：

- Python 运行环境
- Node / npm 运行环境
- Chromium 浏览器
- 公网任务脚本依赖

### 4.4 健康检查与可读报错

为降低运维和排障成本，新增了以下工程化能力：

- 后端在执行公网任务前先访问 `worker` 的 `/health`
- `worker` 返回 Python / Node / 浏览器可用性信息
- 对 `worker` 不可达、解释器缺失、浏览器启动失败、脚本超时等情况提供更明确的报错提示

## 5. 改造后的收益

从答辩视角，这次改造主要体现了以下工程能力：

### 5.1 可复现

同一套代码可以通过 `.env` 与 `docker-compose` 在不同机器上复现运行，不再依赖作者个人电脑环境。

### 5.2 可维护

运行时依赖、脚本执行、结果入库三者职责分离，后续维护时不需要在单一进程里混合处理所有问题。

### 5.3 可扩展

未来如果需要：

- 增加新的公网平台采集脚本
- 横向扩展多个 worker
- 将 worker 部署到独立服务器

都可以在现有架构基础上平滑演进，而不必重写后端任务编排逻辑。

### 5.4 可排障

相比“只在作者电脑上能跑”的实现，当前版本已经可以较清晰地区分：

- 配置错误
- 依赖缺失
- 容器未启动
- 路径挂载错误
- 脚本执行超时

这对于工程项目的实际交付非常重要。

## 6. 如果老师追问“是否已经完全解决”

建议如实回答：

可以说本次已经完成了“从本机硬编码到可配置、可容器化、可排障”的核心改造，显著提升了工程质量与复现能力。

同时也应说明：

- 当前仍需要通过 `.env` 或 Docker 配置正确指定运行参数
- 若更进一步做生产级部署，还可以继续补充更完整的 CI、镜像版本锁定、自动化集成测试与任务队列机制

这样回答的优点是：

- 不夸大完成度
- 明确说明已解决的核心问题
- 展示出对工程边界和后续演进方向的清晰认识

## 7. 一段可直接用于答辩的简述

可以直接概括为：

“项目早期公网任务模块存在本机路径硬编码问题，影响跨环境复现。针对这一点，我做了两层改造：第一层是把 Python、浏览器、Profile 等路径统一改成环境变量配置，并提供自动发现兜底；第二层是把公网任务执行环境从后端中拆分出来，新增独立 worker 容器，在 Docker 中单独提供 Python、Node 和 Chromium 运行时。这样改造后，项目不再依赖作者个人电脑环境，公网任务具备更好的可配置性、可复现性和可维护性，同时也补充了健康检查与可读报错机制，便于部署和排障。”
